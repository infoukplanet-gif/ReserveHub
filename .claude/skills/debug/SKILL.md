# 体系的デバッグ

## Step 1: 症状の特定

「$ARGUMENTS」について調査する。

まず事実を集める:
```bash
# エラーログの確認
# 再現手順の確認
# 期待する動作 vs 実際の動作
```

**推測で動かない。事実だけで判断する。**

## Step 2: 仮説の立案（最大3つ）

エラーから考えられる原因を優先度順に:
1. **最も可能性が高い原因**（理由: ...）
2. **次に可能性が高い原因**（理由: ...）
3. **それ以外**（理由: ...）

## Step 3: 仮説の検証（1つずつ）

各仮説を検証する方法:
```bash
# ログを追加して確認
# 該当コードを読んで確認
# テストを書いて確認
```

**1つ検証して違ったら次へ。複数同時にいじらない。**

## Step 4: 根本原因の特定

5 Whys で根本原因まで掘る:
1. なぜエラーが出た？ → 直接原因
2. なぜその直接原因が存在した？ → ...
3. なぜ... → 根本原因に到達するまで

## Step 5: 修正

- 根本原因を修正する（症状だけ消さない）
- 回帰テストを書く（同じバグが再発しないように）
- 関連箇所に同様の問題がないか確認

## Step 6: 記録

修正内容を簡潔にまとめる:
```
症状: ...
根本原因: ...
修正: ...
再発防止: テスト追加 / ルール追加 / ...
```

## ReserveHub 固有のデバッグチェックリスト

| 症状 | まず確認すること |
|---|---|
| データが見えない | tenant_id のフィルタ漏れ、RLSポリシー |
| 料金がおかしい | pricing_rules の priority、曜日/時間帯の判定 |
| 予約が重複 | 空き枠計算のバッファ時間、トランザクション |
| 回数券が減らない | 消化ロジック、対象メニューの紐付け |
| 認証エラー | Supabase Auth のJWT、app_metadata |
| 画面が真っ白 | Server/Client Component の境界、use client 漏れ |
| ビルドエラー | 型エラー、import パス、missing dependency |

**焦らない。1つずつ。事実ベースで。**
