# Reducing Entropy — 削除バイアス

コードは増える方向に自然に進む。この skill は逆方向に押す。

## 3つの問い

### 1. この問題を解くための最小のコードベースは？
- 14関数ではなく2関数でできないか？
- 0関数（機能自体を消す）でいいのでは？
- 何を消せるか？

### 2. この変更で総コード量は減るか？
- 変更後 > 変更前 なら却下
- 「整理された」が行数増 = エントロピー増
- 「柔軟になった」が行数増 = エントロピー増

### 3. 何を削除できるか？
- この変更で不要になるものは？
- 置き換えたものの残骸は？
- 最大でどこまで消せる？

## レッドフラグ
- 「既存を残して」→ 現状維持バイアス。総量で判断
- 「将来の拡張性」→ YAGNI
- 「関心の分離」→ ファイル数増 = コード増。分離はタダじゃない
- 「型安全のため」→ 何行のコスト？ランタイムチェックの方が少ない場合も

## 計測
変更前後でファイル数・行数を比較する:
```bash
find src/ -name "*.ts" -o -name "*.tsx" | xargs wc -l | tail -1
```

**削除バイアス。最終状態を計測。**
